# Участие в разработке media-downloader.pro

Спасибо, что хотите помочь улучшить **media-downloader.pro**!  
Этот документ объясняет, как правильно предлагать изменения, чтобы их было проще принять и интегрировать.

---

## Как предлагать изменения

Любые изменения (баги, фичи, улучшения) начинаются с обсуждения. Мы используем следующий порядок:

### 1. Issue (проблема или идея)
Перед тем как писать код — создайте **issue**, чтобы обсудить:
- Описание бага или новой функции
- Причины, почему это важно
- Возможные решения

> 🔍 Поиск: проверьте, нет ли уже похожего issue.

Мы используем шаблоны:
- 🐛 **Bug Report** — для ошибок
- ✨ **Feature Request** — для предложений
- 🧰 **Improvement** — для оптимизации

### 2. Обсуждение
Команда ответит в течение **3 рабочих дней**.  
Если идея одобрена — мы помечаем issue как `accepted` и можем назначить вам.

> 💡 Если вы хотите взять задачу — напишите: `Хочу реализовать`.

### 3. Pull Request (запрос на слияние)
После согласования — вносите изменения:

1. Сделайте форк репозитория
2. Создайте ветку: `feature/название` или `fix/описание`
3. Внесите правки
4. Запустите тесты и линтеры (см. ниже)
5. Сделайте коммиты с понятными сообщениями
6. Откройте PR в ветку `main`

---

## Требования к Pull Request

Чтобы PR был рассмотрен, он должен соответствовать этим правилам:

### ✅ Структура
- Ветка создана из актуального `main`
- Изменения минимальны и сфокусированы на одной задаче
- Нет мусора (`.pyc`, `__pycache__`, `.env` и т.п.)

### 📝 Описание PR
Обязательно укажите:
- Ссылку на issue: `Closes #123`
- Что было сделано
- Как проверить изменения (ручное тестирование / запуск)
- Скриншоты (если есть UI)

### ✅ Чек-лист в описании PR
Добавьте чек-лист и отметьте выполненные пункты:

- [ ] Соответствует стилю кода
- [ ] Проходит проверки линтеров
- [ ] Добавлены/обновлены тесты (если нужно)
- [ ] Проверено локально
- [ ] Обновлена документация (если нужно)

---

## Требования к коду

Проект написан на **Python**, используется современный синтаксис (3.9+).

### 🧼 Стиль кода
- Следуем [PEP 8](https://pep8.org/)
- Форматируем через [Black](https://github.com/psf/black) + [isort](https://pycqa.github.io/isort/)
- Типизация: аннотации обязательны
- Документация: docstring по [Google Style](https://google.github.io/styleguide/pyguide.html#38-comments-and-docstrings)

### 🧪 Тесты
- Для новых функций — обязательно добавляйте тесты (`tests/`)
- Используем `pytest`
- Покрытие не должно снижаться

### 🧰 Линтеры
Перед коммитом запустите:

```bash
black .
isort .
flake8
pytest
```

Рекомендуем использовать pre-commit:

```bash
pip install pre-commit
pre-commit install
```

### 📬 Формат коммитов
Используем Conventional Commits:
- feat: добавить загрузку с TikTok
- fix: исправить парсинг URL
- docs: обновить README
- style: форматирование main.py
- refactor: переписать модуль downloader
- test: добавить тесты для YouTube
- chore: обновить зависимости

### Процесс ревью

Когда вы открываете PR:

1. Запускаются CI-проверки (форматирование, тесты)
2. Один из мейнтейнеров назначается на ревью
3. Ответим в течение 3 рабочих дней
4. Укажем замечания — вы вносите правки
5. После одобрения — PR будет слит

*🕒 Если PR простой (мелкий фикс, документация) — может быть принят быстрее.*

Вы можете пинговать мейнтейнера через @dominev, если нет ответа дольше 5 дней.
